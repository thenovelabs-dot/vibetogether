# 앱인토스 알림 플랜

최종 업데이트: 2026-06-17

## 현재 웹 서비스 알림 구조

웹 버전은 두 가지 알림을 사용한다.

- 서비스 내 알림: 상단 벨 드롭다운에서 댓글, 좋아요, 모임 신청, 커피챗 상태를 보여준다.
- 이메일 알림: Supabase Edge Function과 Resend로 모임 댓글/신청 수락/커피챗 관련 이메일을 보낸다.

## 앱인토스 1차 빌드 원칙

토스 로그인 연동 전까지 앱인토스 빌드는 열람 중심으로 운영한다.

- 유지: 서비스 내 알림 UI 자체
- 유지: 게시글 댓글, 프로덕트 댓글, 프로덕트 좋아요 알림
- 제외: 모임 신청/댓글 알림
- 제외: 커피챗 알림
- 보류: 이메일 알림
- 보류: 앱인토스 푸시/토스 알림 연동

현재 코드에서는 앱인토스 타깃일 때 `getNotifications()`가 모임/커피챗 테이블을 조회하지 않도록 분기한다.

## 1단계: 토스 로그인 전

상태: 현재 준비된 앱인토스 안전 빌드

- 비로그인 사용자는 콘텐츠 열람 중심
- 로그인 필요 액션은 `토스 로그인 준비 중` 상태로 비활성화
- 알림 벨은 로그인 세션이 없으면 보이지 않는다.
- 이메일 알림은 앱인토스 사용자에게 발송하지 않는다.

이 단계에서는 심사 리스크를 줄이는 것이 목적이다. 알림 기능을 과하게 노출하지 않는다.

## 2단계: 토스 로그인 연동 후

토스 로그인과 Supabase 유저 매핑이 끝나면 서비스 내 알림을 앱인토스 타깃에서도 활성화한다.

대상 이벤트:

- 내 게시글에 댓글
- 내 게시글 댓글에 대댓글
- 내 프로덕트에 댓글
- 내 프로덕트 댓글에 대댓글
- 내 프로덕트 좋아요

제외 이벤트:

- 모임 신청
- 모임 댓글
- 커피챗 신청/수락

구현 기준:

- 앱인토스 타깃에서는 `features.meetups=false`, `features.coffeechat=false`를 유지한다.
- `getNotifications(userId, { includeMeetups: false, includeCoffeechat: false })`를 사용한다.
- 알림 클릭 링크가 `/meetup*`이면 `/home`으로 보내는 방어 로직을 유지한다.

## 3단계: 앱인토스 알림/푸시 검토

앱인토스 SDK/API에서 사용자에게 알림을 보낼 수 있는 공식 수단이 확인된 뒤 검토한다.

검토할 것:

- 토스 로그인 사용자 식별자와 알림 수신자 매핑
- 사용자의 알림 수신 동의 UX
- 알림 발송 API의 서버 인증 방식
- mTLS 또는 서버 간 인증 필요 여부
- 발송 가능한 알림 유형과 심사 정책

앱인토스 푸시나 토스 알림은 클라이언트에서 직접 호출하지 않는다. 필요한 경우 서버 또는 Supabase Edge Function 뒤에 둔다.

## 이메일 알림 정책

앱인토스 타깃에서는 이메일 알림을 기본 비활성으로 둔다.

이유:

- 토스 로그인은 이메일을 필수로 보장하지 않을 수 있다.
- 앱인토스 내 사용자는 토스 앱 안에서 완결되는 경험을 기대한다.
- 외부 이메일 기반 연결은 커피챗/모임 기능과 결합될 경우 만남성 리스크가 커진다.

이메일 알림을 다시 켜려면 다음 조건을 먼저 만족해야 한다.

- 토스 로그인 후 이메일 수집/이용에 대한 명확한 동의
- 개인정보 처리방침 업데이트
- 수신 거부 또는 알림 설정 동선
- 이메일이 앱인토스의 핵심 흐름을 외부로 빼내지 않는다는 검토

## 운영 메모

- 앱인토스 1차 출시 설명에는 "알림"을 핵심 기능으로 적지 않는다.
- 콘솔의 앱 내 기능 설명은 "게시판과 프로덕트 탐색/작성" 중심으로 유지한다.
- 알림은 로그인 후 보조 기능으로 다룬다.
- 모임/커피챗이 다시 들어오기 전까지 사람 간 직접 연결 알림은 만들지 않는다.
